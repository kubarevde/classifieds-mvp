## Опорные точки архитектуры (менять осторожно)

1. **Разделение «контракт vs реализация» в `src/services/*/types.ts` + `mock.ts`** — это главный шов для подключения бэкенда; его стоит сохранять и расширять, а не пробивать прямыми `fetch` из компонентов.
2. **`src/entities/*`** — хорошая база для shared types между фронтом и API; поддерживать без React-импортов.
3. **`src/api/contracts/*`** — готовый задел под SDK; синхронизировать с OpenAPI/сервером при появлении API.
4. **Корневой composition root** — `src/app/layout.tsx` + провайдеры; изменения здесь затрагивают всё приложение.
5. **Политика тарифов** — `entities/billing` + `feature-gate` + `SubscriptionProvider`; при появлении реальной подписки переносить решения на сервер, UI оставить тонким.

---

## Самые опасные места

| Риск | Почему |
|------|--------|
| **Отсутствие серверной авторизации** | Любой клиентский гейт обходим |
| **Монолитные клиентские модули** | Регрессии при изменении одной фичи ломают несколько сценариев |
| **Двойной путь данных (lib vs services)** | Ошибки синхронизации, дублирование логики фильтрации |
| **Нет наблюдаемости** | Прод-инциденты будут невидимы до жалоб пользователей |
| **Нет настоящей БД** | Любая «сделка» по миграции потребует проектирования с нуля параллельно UI |

## Модули, проще всего расширять

- **Чистые функции** в `src/lib/listings.filters.ts`, `src/services/risk/rules.ts`, `src/services/auctions/rules.ts` — покрыты тестами или легко покрываются.
- **Feature gate** — уже выделенный сервис + тесты; проще добавить новый `Feature` и правило.
- **Новые маршруты App Router** — конвенция Next ясна; мало риска для изолированных read-only страниц.

## Идеи рефакторинга перед ростом

1. **Завершить миграцию чтения листингов** на единый вход (`ListingsService` только), `lib/listings.data` оставить внутренностью сервиса.
2. **Разрезать крупные `*page-client.tsx`** на подкомпоненты + хуки `useListingsPageModel` и т.п. с явными типами состояния.
3. **Вынести `StoreSubscriptionTierId` и родственные типы** из `store-dashboard-shared` в `entities` или `services/billing`, чтобы `entitlements/config` не зависел от UI.
4. **BFF слой** (Route Handlers или отдельный сервис) при появлении секретов и авторизации — не светить токены в браузер лишний раз.
5. **Микрофронт или отдельное приложение** для `/admin` при найме отдельной команды trust & safety (опционально, на средний горизонт).

## Какие фичи добавить проще всего сейчас

- **Новые read-only страницы** на существующих моках (документация, маркетинговые лендинги).
- **Ещё один «мир» или категория** в discovery — правки данных и фильтров.
- **Новый флаг в `Feature` + отображение в UI** — цепочка уже проложена через feature-gate.
- **Дополнительные поля в черновике объявления** — инфраструктура Zustand persist уже есть.

## Первоочередной порядок работ (рекомендация аудита)

1. **Зафиксировать контракт первого реального API** (листинги или auth) и добавить `http.ts` + фабрику выбора mock/http по env — см. шаги в `ARCHITECTURE.md`.
2. **Ввести серверную сессию/JWT** и защиту `/admin` через `middleware` или серверные layout-guards с валидацией на бэкенде.
3. **Синхронизировать документацию** (README vs Zustand).
4. **Подключить мониторинг** (Sentry или аналог) вместо TODO в `monitoring.ts`.
5. **Расширить тесты** на `requests` service и критичные фильтры дашборда.
